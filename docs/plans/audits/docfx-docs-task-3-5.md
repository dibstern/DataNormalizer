# Audit: Tasks 3-5 — Content and Gitignore

**Summary:** Tasks 3-5 are well-structured and closely aligned with actual README content. The `_layout: landing` frontmatter is valid for the modern template. The `.gitignore` patterns correctly exclude generated files while preserving the hand-written `docs/api/index.md`. One substantive issue found: `docs/plans/` exclusion from DocFX relies on implicit glob behavior but is never explicitly stated in the `docfx.json` `exclude` list — this is fragile if content patterns change. A few minor line-reference inaccuracies exist but won't affect implementation.

## Findings

### Finding 1: `_layout: landing` frontmatter is valid
**Action:** Accept
**Details:** The plan uses `_layout: landing` in `docs/index.md` YAML frontmatter. The DocFX modern template documentation explicitly lists `_layout` as a supported metadata property with `landing` and `chromeless` as valid values. This is correct and will render a landing page without the TOC sidebar. No issue here.

### Finding 2: `docs/plans/` exclusion from DocFX build is implicit, not explicit
**Action:** Amend Plan
**Details:** The design document states "The existing `docs/plans/` directory is internal and will not be published" but the `docfx.json` content patterns in Task 2 rely on `*.md` (non-recursive glob) to avoid including `plans/`. While `*.md` does NOT match `plans/**/*.md` in DocFX's glob implementation, this is fragile — if anyone later changes `*.md` to `**/*.md`, or adds a catch-all content entry, the plan files (including implementation details, audits, design docs) would be published to the public documentation site.

**Recommendation:** Add an explicit `exclude` to the content section in Task 2's `docfx.json`:
```json
{
  "files": ["articles/**.md", "articles/**/toc.yml", "toc.yml", "*.md"],
  "exclude": ["plans/**"]
}
```
This makes the exclusion explicit and resilient. Since this is a Task 2 issue surfaced during Task 3-5 audit, it should be flagged as a Task 2 amendment.

**File:** `docs/plans/2026-03-20-docfx-docs-implementation.md:73` (Task 2, `docfx.json` content section)

### Finding 7: `.gitignore` doesn't cover `docs/api/toc.yml` (auto-generated by DocFX metadata)
**Action:** Amend Plan
**Details:** When `docfx metadata` runs, it generates YAML files in `docs/api/` including `toc.yml`. The proposed `.gitignore` pattern `docs/api/*.yml` DOES match `toc.yml`, so this is actually covered. However, DocFX metadata also generates files in nested subdirectories for namespaces (e.g., `docs/api/DataNormalizer.Runtime.yml`). The `*.yml` pattern only matches one directory level deep. If DocFX generates deeper paths, they'd leak through.

In practice, DocFX `mref` output format places all `.yml` files flat in the `api/` directory (no subdirectories), so `docs/api/*.yml` is sufficient. This is an informational note only.
